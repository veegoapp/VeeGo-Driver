You are working on a React Native Expo driver app.

We have a critical architectural issue related to identity isolation and onboarding state persistence.

CURRENT PROBLEM:
The app currently stores driver onboarding state using a device-level key:

- veego_service_type is stored in AsyncStorage (device-wide)
- It is NOT tied to a user account or driver identity
- This causes cross-account leakage on shared devices

CRITICAL BUG SCENARIO:
1. Driver A logs in on a device
2. Driver A selects a service (e.g. CAR)
3. veego_service_type = "CAR" is saved in AsyncStorage
4. Driver A logs out
5. Driver B signs up or logs in on SAME device
6. Driver B bypasses service-select and is incorrectly routed directly to /(tabs)
   → because veego_service_type already exists from Driver A

This breaks onboarding correctness and violates per-account isolation.

ADDITIONAL ISSUE:
Onboarding state is implicitly derived ONLY from:
- existence of veego_service_type

There is NO explicit per-user onboarding flag, such as:
- onboardingCompleted
- serviceSelected per account
- driverProfile.serviceType stored server-side

GOAL:
Move onboarding + service selection state from device-based storage to account-based identity.

REQUIREMENTS:

1. Replace device-scoped storage logic:
   - STOP relying solely on AsyncStorage key veego_service_type for routing decisions
   - DO NOT allow cross-account reuse of service selection state

2. Introduce user-scoped service selection logic:
   - Bind service selection to authenticated user identity (userId from auth token or session)
   - Preferred approach:
     store service type in a structure like:
       { userId: serviceType }

   OR retrieve it from backend profile if available

3. Fix post-auth routing logic:
   - navigateAfterAuth must NOT trust raw AsyncStorage value alone
   - It must validate that stored service_type belongs to current user
   - If mismatch OR missing → route to /service-select

4. Ensure correct onboarding behavior:
   - First login per account → must always see /service-select
   - Returning same account → skip if service already belongs to that account
   - Different accounts on same device must NOT share onboarding state

5. Maintain backward compatibility:
   - If legacy single string exists in AsyncStorage, treat it as invalid unless tied to current user
   - Migrate safely without breaking existing users

EXPECTED RESULT:

- Each driver account has independent onboarding state
- No cross-account service leakage
- Service selection becomes identity-based, not device-based
- Login flow becomes deterministic and secure per user

IMPORTANT:
- Do NOT break existing auth_token / refresh_token system
- Do NOT change backend APIs unless necessary
- This is a frontend architecture + storage isolation fix
