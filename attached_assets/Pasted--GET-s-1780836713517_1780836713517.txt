تعليمات الفرونت اند — ما الذي تغيّر في الباك اند وكيف تتعامل معه
التعديل الذي تم في الباك اند
GET /shuttle/timeslots/:routeId — تم تحديثه بالكامل.

1. اختيار الأسبوع (Select Work Week)
لا تغيير مطلوب من الفرونت هنا.

الفرونت يولّد الأسابيع الثمانية محلياً (Sun–Thu) وهذا صحيح ومقبول. الباك اند يقبل أي weekStart يكون Sunday ومن الأسبوع القادم فأكثر.

القاعدة التي يجب أن يعرفها الفرونت:

weekStart = دائماً Sunday بصيغة YYYY-MM-DD
weekEnd = الخميس بعده بـ 4 أيام
لو أرسل الفرونت يوم مش Sunday → الباك اند يرجع 400
2. اختيار وقت المغادرة (Departure Time) — التعديل الرئيسي
الـ Endpoint
GET /api/shuttle/timeslots/:routeId?weekStart=YYYY-MM-DD

متى تستدعيه
استدعه فور ما السائق يختار أسبوعاً — أرسل weekStart الأسبوع الذي اختاره السائق.

// السائق اختار أسبوع Jun 15–19
GET /api/shuttle/timeslots/4?weekStart=2026-06-15

بدون weekStart → الباك اند يرجع بيانات الأسبوع القادم افتراضياً — لا تعتمد على ذلك، أرسله دائماً.

الـ Response الكامل
{
  "routeId": 4,
  "routeName": "Ain Shams → El Maadi #1",
  "weekStart": "2026-06-15",
  "weekEnd": "2026-06-19",
  "data": [
    {
      "id": 4,
      "departureTime": "14:00",
      "availableSeats": 7,
      "totalSeats": 14,
      "isBooked": false,
      "isTaken": false
    },
    {
      "id": 5,
      "departureTime": "08:00",
      "availableSeats": 0,
      "totalSeats": 14,
      "isBooked": true,
      "isTaken": false
    }
  ],
  "total": 2
}

شرح كل حقل في data[*]
الحقل	النوع	المعنى	كيف يستخدمه الفرونت
id	number	الـ ID المُرسل في POST /shuttle/route-bookings كـ timeSlotId	أرسله كما هو في الحجز
departureTime	string	وقت المغادرة بصيغة "HH:MM" (24 ساعة)	اعرضه كـ label للـ chip
availableSeats	number | null	عدد المقاعد المتاحة لهذا الأسبوع تحديداً	اعرض badge بالعدد. لو null = لا يوجد رحلة مجدولة لهذا الأسبوع
totalSeats	number | null	سعة المركبة	لو null = نفس سبب availableSeats
isBooked	boolean	السائق الحالي حجز هذا الـ slot لهذا الأسبوع	true → لوّن الـ chip وضع "محجوز منك"
isTaken	boolean	سائق آخر حجز هذا الـ slot لهذا الأسبوع	true → الـ chip غير متاح (disable + لون مختلف)
منطق عرض الـ Chip
isBooked = true  → chip مفعّل، مميّز (لون مختلف)، لا يُسمح بإعادة الحجز
isTaken  = true  → chip معطّل (disabled)، لا يستطيع السائق حجزه
كلاهما false     → chip متاح للحجز
availableSeats = null → لا تعرض badge المقاعد (أو اعرض "—")
availableSeats = 0    → يمكن عرض "ممتلئ" لكن الحجز قد يكون مسموحاً (حجز السائق ≠ حجز الركاب)

مثال استدعاء كامل في التطبيق
// عندما يختار السائق أسبوعاً
const fetchTimeslots = async (routeId, weekStart) => {
  const response = await fetch(
    `/api/shuttle/timeslots/${routeId}?weekStart=${weekStart}`,
    { headers: { Authorization: `Bearer ${token}` } }
  );
  const json = await response.json();
  return json.data; // المصفوفة المباشرة
};
// عند الحجز
const createBooking = async (routeId, timeSlotId, weekStart) => {
  const response = await fetch('/api/shuttle/route-bookings', {
    method: 'POST',
    headers: {
      Authorization: `Bearer ${token}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({ routeId, timeSlotId, weekStart }),
  });
  return response.json();
};

ملاحظة مهمة على availableSeats = null
لو الـ route ليس عنده trips مجدولة في الأسبوع المختار → availableSeats وtotalSeats يكونوا null. هذا طبيعي لأن المقاعد مرتبطة بالـ trips وليس بالـ slots مباشرة. الفرونت يتعامل مع null بأحد الطريقتين:

يخفي الـ badge ولا يعرض رقم
يعرض "—" بدل الرقم
الحجز نفسه يظل مسموحاً حتى لو availableSeats = null.

ملخص التعديلات المطلوبة في الفرونت فقط
#	التعديل
1	عند اختيار أسبوع → استدعِ GET /shuttle/timeslots/:routeId?weekStart=YYYY-MM-DD من جديد
2	اقرأ البيانات من response.data (مصفوفة)
3	استخدم slot.id كـ timeSlotId في الحجز
4	استخدم slot.isBooked لتمييز ما حجزه السائق نفسه
5	استخدم slot.isTaken لتعطيل الـ chip لو حجزه سائق آخر
6	تعامل مع availableSeats = null بإخفاء الـ badge
