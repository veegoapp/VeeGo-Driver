You are working on a React Native Expo driver app with a real-time service control system.

We already fixed:
- app startup routing (Phase 1)
- identity isolation per user (Phase 2)

NOW we are fixing the final and most critical layer:
SERVICE CONTROL SAFETY + CONSISTENCY.

CURRENT PROBLEM:
The service control system can still incorrectly allow services to appear or behave as "available" in edge cases.

ROOT ISSUES:

1. FAIL-OPEN BUG (CRITICAL)
In multiple places:
if service control data is missing or fails to load → services default to OPEN / AVAILABLE

Example pattern:
if (!ctrl) return OPEN;

This causes:
- disabled services appearing active
- coming_soon services becoming clickable during network failure
- temporary backend failures unlocking all services

2. RACE CONDITION BETWEEN STATES
There is a timing gap:
- Auth resolves
- ServiceControlProvider is still loading
- UI renders with services = []

During this window:
→ all services appear enabled or interactive

3. SOCKET VS REST DESYNC
- REST fetch may fail or be delayed
- socket updates may arrive later
- UI briefly shows outdated OPEN state

4. WEAK GUARD LAYER
- useServiceGuard is only a UI overlay
- services are still technically accessible through tap handlers during edge cases

GOAL:
Make service control system FAIL-SECURE, not FAIL-OPEN.

REQUIREMENTS:

1. HARD FAIL-SECURE DEFAULT
Replace ALL fallback logic like:

if (!ctrl) return OPEN;

WITH:

if (!ctrl) return BLOCKED;

No service can ever default to OPEN unless explicitly confirmed by backend.

2. INTRODUCE GLOBAL "SERVICE LOADING LOCK"
While ServiceControlProvider is loading:
- ALL services must be treated as BLOCKED or DISABLED
- UI must NOT allow interaction (no taps, no navigation)

3. UNIFY REST + SOCKET STATE
Ensure:
- REST is initial source of truth
- socket only applies patches
- socket cannot override loading state or reset services to OPEN

4. CENTRALIZE SERVICE STATE DECISION
All service availability must go through ONE function:
getServiceStatus(serviceType)

This function must:
- never return OPEN on missing data
- always prioritize explicit backend state
- enforce BLOCKED during loading or error states

5. STRICT TAP PROTECTION
Ensure handleServiceTap or equivalent:
- checks loading state first
- checks service existence
- checks isEnabled
- checks displayMode
BEFORE allowing ANY navigation or callback

6. ERROR STATE = SAFE STATE
If:
- API fails
- socket disconnects
- partial data exists

THEN:
System must show services as DISABLED / BLOCKED, NOT OPEN.

EXPECTED FINAL BEHAVIOR:

- If backend is down → NO services are clickable
- If data is loading → NO services are clickable
- If socket is delayed → no temporary OPEN state ever appears
- Only explicit backend-approved services can be used

IMPORTANT:
- Do NOT break auth system
- Do NOT change backend APIs
- This is strictly a frontend safety + consistency layer fix

Deliver a production-grade fail-safe service control system.
